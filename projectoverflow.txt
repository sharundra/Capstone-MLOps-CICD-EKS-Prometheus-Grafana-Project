--------------------------Setting Up project structure--------------------------
1. Create repo in remote and then clone it in local
2. Create a virtual environement -- conda create -n <venv name> python=3.10
    a. Activate the virtual environment -- conda activate <venv name>
3. pip install cookiecutter
4. Run this command to scaffold project structure  -- cookiecutter -c v1 https://github.com/drivendata/cookiecutter-data-science
5. Copy the contents of newly created folder(by cookiecutter) into root folder and delete the empty cookiecutter folder.
6. Rename src.models -> src.model
    a. Also, bring the correct requirements.txt and run, "pip install -r requirements.txt"

--------------------------  Setup MLFlow on Dagshub------------------------------
7. Go to https://dagshub.com/dashboard and connect a git repo to dagshub -- Connect -> new repo -> connect a repo -> only selected repos -> select a repo -> save -> connect the repo
8. Copy experiment tracking URL and code snippet given there (can try MLFlow UI)
9. pip install dagshub and mlflow

10. Run experiment notebooks
11. git add -> commit -> push

12. For data versioning -- dvc init 
13. Create a local folder for temprory work as 'local_s3'
14. Run this command -- dvc remote add -d mylocal local_s3

--------------------------  Pipeline training and versioning ------------------------------
15. Add code to below files/folders inside src dir --
    a. logger
    b. data/data_ingestion.py :
        i.  run and test it using direct link for data access -- train and test data should appear in data/raw folder.
        ii. then bring "connections" folder inside src and enable s3 data access in data_ingestion.py (using boto3) and run it to verify if s3 connection works properly.
        iii. Once s3 connection testing done, revert back to direct data link for data acccess and remove aws credentials from the code.
    c. data/data_preprocessing.py -- run it and then it should save preprocessed data into data/interim folder.
    d. features/feature_engineering.py -- run it and vectorzed data should go to data/processed folder and also vectorizer pickle model should be saved into "models" folder.
    e. model/model_building.py -- run it and model.pkl should be saved at "models" folder 
    f. model/model_evaluation.py :
        i. comment out the product code piece and uncomment local run piece. Give the correct ml_tracking_uri and then correct repo_owner and repo_name for dagshub_init and then run the code. "metrics.json" would be saved inside "reports" folder. Go to newest experiment in mlflow UI to track this run there. 
    g. model/register_model.py :
        i. again comment out production connection code piece and uncomment local dagshub connection piece. Then run and version 1 of model gets registered in MLflow.
        ii. TO register version 2 of the model also in MLflow -- Change some hyperparameter in model_building.py and again run model_building.py -> model_evaluation.py -> register_model.py

16. Add dvc.yaml file which should contain stages/instructions till model_registration
    a. This yaml file is to automate the pipeline from data ingestion till model registration.
17. Add params.yaml file --> to aggregate all the parameters from all the scripts here only.
    a. Uncomment the params.yaml related parts in scripts to access parameter values from param.yaml (here data_ingestion.py and feature_engineering.py)
    b. We are adding this params.yaml now because dvc.yaml talks about commands, dependencies, params and outputs for each stage(data ingestion, data preprocessing, model building etc.).

18. Now dvc pipeline is ready to run :
    a. run "dvc repro" --> It will run the entire pipeline till model registration and would save version 3 of model in MLflow.

19. Check/delete existing local dvc remote :
    a. dvc remote list 
    b. dvc remote remove <name_of_remote>
20. Set aws credentials --> run "aws configure" and put aws access key id and secret access key
21. Add s3 bucket as dvc remote storage :
    a. run "dvc remote add -d myremote s3://<bucket-name>" -- we can cross-check if it was run successfully at .dvc/config, where s3 url would show up now.
22. Push data to s3 bucket :
    a. dvc status -- will be up to date as we have already run "dvc repro"
    b. dvc commit
    c. dvc push
    d. change one of the parameters in params.yaml --> run "dvc status" as it will show changes status --> run "dvc repro" and new version 4 of model will be regsitered in MLflow registry --> again repeat "dvc status" which will be up to date now then "dvc commit" then "dvc push"

----------------------------------  CICD pipeline  -------------------------------------
23. Bring the new folder, flask_app and have relevant files and folders inside it.
    a. Inside the app.py, comment production code piece and use local use code piece and then run this file --> python flask_app/app.py (make sure stage is "staging" and not "production" for now) -- latest model will get served from MLflow to here and for any new text, prediction will be rendered on the UI runnning on port localhost:5001
    b. app.py has 3 routes, /, predict and metrics so once app is live on localhost, check all 3 routes
24. Add .gihub/workflows/ci.yaml file 
    a. Generate one authentication token from dagshub -- Your Settings --> Tokens --> Generate new Tokens
    b. Put that dagshub toekn in github secrets -- settings --> secrets and variables --> actions 
    c. Update the scripts to use production code wherever local code for dagshub/mlflow access was being used.
    d. Git add, commit and push again and check the CI progress in github actions 
    e. After successful run of CI pipeline in Github actions, again it can be sone, dvc status, dvc commit and dvc push.

25. Bring scripts and tests folder. Update the production code related dagshub variables.
    a. Update ci.yaml for model testing, promotion to production and flask app testing.
    b. git add, commit and push and follow github actions for ci pipeline.


----------------------------------  Moving to Docker and EKS  -------------------------------------
26. Generate one optimized requirements.txt inside folder flask_app
    a. "pip install pireqs"
    b. "cd flask app" then run "pipreqs . --force" -- It does this, in the current folder (.) scan all the python import statements and generate requirements.txt
27. Add Dockerfile and start docker-desktop in background 
28. Run the command from the root directory -- "docker build -t capstone-project-app:latest ."
    a. Run "docker run -p 8888:5001 -e CAPSTONE_TEST=<generated dagshub token> capstone-project-app:latest" -- To tackle dagshub authentication iussue, we are using porduction grade code and passing the token while running the docker image.

29. If not done already (because we have done, using command, "aws configure"), set up AWS services for below variables --
    a. AWS_ACCOUNT_ID (click on root id name in a small tab in top right, copy account id from there); AWS_ACCESS_KEY_ID; AWS_SECRET_ACCESS_KEY; AWS_REGION; ECR_REPOSITORY
    b. Once AWS setup is done, add these variables and their values in github actions' settings/secrets and variables/actions
    c. Also, add this permission to the IAM user: AmazonEC2ContainerRegistryFullAccess -- IAM -> IAM users -> select correct user -> add permissions -> attach policies directly -> select policy name -> next -> done.
30. Execute the pipeline till the stage where we build and push the image to ECR.


31. Setup required before moving to EKS deployment --
    a. Verify AWS CLI using this command --> aws --version 
    b. Verify kubectl using this command --> kubectl version --client
    c. verify eksctl using this command --> eksctl version 
        i. All three mentioned above can be very easily installed on mac using brew commands, i.e., "brew install awscli", "brew install kubectl", "brew install eksctl".
    d. Now since awscli and eksctl are installed and "aws configure" command has already run, confirm if the setup is correct without actually spending money on a cluster by running below command:
        i. run "eksctl get clusters" --> if it returns "No clusters found" that means setup is good. 
32. Create a cluster on EKS using this command --> "eksctl create cluster --name flask-app-cluster --region us-east-1 --nodegroup-name flask-app-nodes --node-type t3.small --nodes 1 --nodes-min 1 --nodes-max 1 --managed"
33. Update kubectl Config. Once the cluster is created, eksctl will automatically update our kubectl config file. However, we can verify and set it manually using the commnad :
    a. "aws eks --region us-east-1 update-kubeconfig --name flask-app-cluster" --> This ensures that our kubectl is pointing to the correct cluster.
34. Check EKS cluster configuration. Run this command, "aws eks list-clusters" and ensure we can access the eks cluster.
    a. Optional command to delete the cluster --> "eksctl delete cluster --name flask-app-cluster --region us-east-1"
    b. Then verify the deletion --> "eksctl get cluster --region us-east-1"
35. Verify the cluster status :
    a. Run the command, "aws eks --region us-east-1 describe-cluster --name flask-app-cluster --query "cluster.status""
36. Check if kubectl has connectivity with cluster nodes --> run this command, "kubectl get nodes"
37. Check the namespaces --> "kubectl get namespaces"
38. Verify our deployment (if there are any) --> "kubectl get pods" then to check services, run "kubectl get svc"
39. Deploy the app on EKS via CICD pipeline :
    a. Edit ci.yaml, deployment.yaml, Dockerfile
    # Read a little about AWS CloudFormation service and how it is related to EKS -- When we run commands to create an EKS cluster, CloudFormation creates a stack in the backgroud regarding what all is required to create a cluster and does the required job in the background (like creating an EC2 instance in the background which we can verify on CloudFormation and EC2). 
        i. In case if cluster is down in future for some reason, this stack helps AWS bring back cluster back to action again.
    b. Now edit the security group for nodes and edit inbound rule for 5001 port
        i. So as one Ec2 insance/node is already created by CloudFormation in the background as per the given command, go to that instance -> instance id -> security -> security groups -> Inbound rules -> edit inbound rules -> add rule (port range as 5001 and source as "anywhere IPv4") -> Save.
    c. Now do git add, commit and push to trigger cicd for eks deployment.
40. Once the deployment is successful, 
    a. Verify the deployment, run "kubectl get pods" and we should get STATUS as RUNNING.
    b. get the load balancer URL --> "kubectl get svc flask-app-service"
